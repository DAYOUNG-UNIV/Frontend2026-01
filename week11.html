WBS Guide

컴포넌트를 태스크(Issue)로 쪼개는 3단계 공식

"로그인 창 만들기"처럼 뭉뚱그려 적으면 일을 나눌 수도, 진행 상황을 파악할 수도 없습니다.
하나의 덩어리(컴포넌트)는 보통 아래 3가지 관점의 상세 태스크로 분해해야 합니다.

1. UI 퍼블리싱 (보이는 것)

HTML 뼈대를 잡고 Tailwind CSS로 디자인을 입히는 작업입니다. 모바일/데스크탑 화면 크기에 맞추는 반응형 처리도 여기에 포함됩니다.

태스크 작성 예시: GNB 데스크탑/모바일 반응형 UI 퍼블리싱

2. 인터랙션 & 로직 (움직이는 것)

JavaScript(또는 React)를 활용해 클릭 이벤트, 팝업 애니메이션, 입력 폼 검증(Validation), 상태(State) 관리 등을 구현하는 작업입니다.

태스크 작성 예시: 햄버거 버튼 클릭 시 모바일 사이드메뉴 토글 로직 구현

3. 데이터 연동 (연결되는 것)

하드코딩된 가짜 글자를 진짜 데이터로 바꾸는 작업입니다. 프론트 단에서의 가짜 데이터(Mock Data) 바인딩, 외부 API 호출, 라우팅 처리가 포함됩니다.

태스크 작성 예시: 구글 Auth 연동 및 헤더에 유저 프로필 정보 렌더링

쪼개진 컴포넌트들을 GitHub Projects에 태스크(Issue)로 등록하고 관리하는 방법을 순서대로 알아봅시다.

Why Kanban?

왜 하필 '칸반(Kanban)'인가요?

칸반은 공장 생산 라인의 '작업 지시서(포스트잇)'를 벽에 붙여 관리하던 것에서 유래한 가장 직관적인 시각적 협업 도구입니다. 깃허브의 수많은 프로젝트 템플릿 중 우리가 이것을 선택하는 이유는 분명합니다.

  • 한눈에 파악: 누가 어떤 일을 하고 있고, 무엇이 막혀있는지(Bottleneck) 직관적으로 보입니다.
  • WIP(진행 중인 작업) 제한: 한 사람이 너무 많은 일을 동시에 벌여 집중력이 분산되는 것을 막아줍니다.
  • Git 완벽 연동: 코드를 PR하고 병합(Merge)하면, 카드가 알아서 'Done' 칸으로 이동합니다!

Issue와 Project, 무슨 차이인가요?

Issues (개별 작업)

"무엇을 해야 하는가?"
각각의 할 일, 버그 신고, 기능 추가 요청 등을 적어두는 개별적인 문서(포스트잇 1장)입니다.

  • 로그인 버튼 색상 변경
  • 지도 API 키 발급받기

Projects (거시적 관리)

"전체 흐름은 어떤가?"
산발적인 수십 개의 Issue들을 한곳에 모아, 작업의 진행 상태(Todo, In Progress)를 관리하는 거대한 현황판입니다.

Issue A Issue B → In Progress
결론: 먼저 Issue 탭에서 할 일을 모두 쏟아내고(백로그), Projects 탭에서 그걸 모아 관리하는 것이 정석입니다!

1 백로그(Backlog) 쌓기: 모든 태스크를 Issue로 등록

📚 백로그(Backlog)란 무엇인가요?

앞서 WBS를 통해 화면을 쪼개고 수많은 태스크를 도출했습니다. 이 아이디어와 할 일들을 기억력에 의존하지 않고, 하나도 빠짐없이 깃허브 Issue에 모두 적어두는 '거대한 장바구니(창고)'를 백로그라고 부릅니다.

  • 우선순위 상관없이 기획된 모든 기능을 다 밀어 넣습니다.
  • 그리고 이 수많은 백로그 중에서 13주차 테스트를 위해 당장 이번 주에 개발할 핵심 기능에만 MVP 라벨을 붙입니다!

이슈 1개는 도대체 어느 정도로 쪼개야 하나요?

피해야 할 작성법 (단위가 너무 큼)
메인 페이지 전체 만들기

이슈가 너무 크면 혼자 며칠을 붙잡고 있어야 합니다. 나중에 PR을 올리면 코드가 수백 줄에 달해 팀원이 리뷰를 진행하기가 매우 어려워집니다.

권장하는 작성법 (적절한 단위)

Step 3

AI와 최고의 시너지, Tailwind CSS 완벽 이해 🌬️

처음 웹 개발을 시작할 때, 프레임워크(Framework)란 무엇일까요?
어렵게 생각할 필요 없습니다. "자주 쓰는 코드를 미리 만들어둔 똑똑한 레고 상자"입니다.

과거: 맨땅에 집 짓기 (순수 CSS)

나무를 직접 베어오고, 벽돌을 하나하나 구워서 집을 지어야 했습니다. 버튼 하나를 예쁘게 만들려면 CSS 파일에 수십 줄의 코드를 직접 고민하며 써야 했죠.

프레임워크란? (밀키트/이케아)

미리 썰어진 야채와 소스가 담긴 '밀키트'나, 설명서대로 조립만 하면 되는 '이케아 가구'와 같습니다. 전 세계 천재 개발자들이 "이 정도 디자인은 미리 만들어둘게, 넌 조립만 해!" 하고 제공하는 도구입니다.

왜 하필 Tailwind 인가요?

수많은 프레임워크 중에서도, HTML 태그 안에 클래스 이름(예: bg-blue-500)만 적으면 바로 디자인이 완성되는 가장 직관적인 도구입니다. 특히 ChatGPT가 이러한 방식의 코드를 매우 정확하게 작성해 줍니다!

코드로 비교해 볼까요?

과거 (HTML + CSS 분리)
<button class="my-custom-btn">클릭</button>
.my-custom-btn {
  background-color: #3b82f6;
  color: white;
  border-radius: 8px;
  padding: 8px 16px;
}
파일 왔다 갔다 하기 귀찮고, 이름 짓기 힘들다.
현재 (Tailwind CSS)
<button class="bg-blue-500 text-white rounded-lg px-4 py-2">
  클릭
</button>
클래스 이름이 곧 디자인! 직관적이고 빠르다.

Vibe Design과 DESIGN.md

말로 코딩하는 '바이브 코딩(Vibe Coding)' 시대! 하지만 팀원 4명이 각자 AI에게 화면을 맡기면 어떻게 될까요? 버튼 색깔과 둥글기가 제각각인 일관성이 완전히 무너진 '프랑켄슈타인 UI'가 만들어집니다.

해결책: Google Stitch & DESIGN.md
최근 구글 Labs에서 발표한 AI 디자인 툴 'Stitch'는 이 문제를 DESIGN.md라는 마크다운 파일 하나로 해결했습니다. AI에게 "우리 팀 디자인 규칙은 이거니까 이 안에서만 짜!"라고 한계선을 그어주는 AI 전용 디자인 시스템입니다.

우측의 DESIGN.md 템플릿을 복사해서 팀 레포지토리에 넣거나, AI에게 질문할 때 맨 위에 항상 붙여넣으세요! (Tailwind와 찰떡궁합입니다)

DESIGN.md (팀 공통 시스템 프롬프트) 
# AI Agent Context: DESIGN.md
You are generating UI code. You MUST strictly follow the design tokens below using only Tailwind CSS. Do not invent new colors or spacing.

## 1. Brand Colors
- Primary: `bg-indigo-600`, `text-indigo-600` (Hover: `indigo-700`)
- Secondary: `bg-slate-800`
- Background: `bg-slate-50`
- Text: `text-slate-800` (Headers), `text-slate-500` (Body)

## 2. Typography
- Font Family: `font-sans` (Pretendard)
- Headings: `font-extrabold`, `tracking-tight`

## 3. Shapes & Shadows
- Global Border Radius: `rounded-2xl` for large cards, `rounded-lg` for buttons.
- Shadows: `shadow-md` (Default), `shadow-xl` (Hover state)
- Transition: Always add `transition-all duration-300` for interactive elements.

## 4. Layout
- Max Container Width: `max-w-6xl mx-auto`
- Grid/Flex: Strongly prefer CSS Grid for lists (`grid grid-cols-1 md:grid-cols-3`)

껍데기를 넘어 진짜 서비스로! (페이지 이동 & 데이터) 🚀

"버튼을 누르면 다음 화면으로 어떻게 넘어가나요?", "내가 쓴 글은 어디에 저장되나요?"
어려운 백엔드 서버 없이도, 프론트엔드 기술만으로 이 문제들을 해결하는 핵심 기술 두 가지를 알아봅시다.

Q1. 화면 이동은 어떻게 하나요?

방식 A HTML 파일 여러 개 만들기 (전통적)

index.html, detail.html 등 파일 자체를 여러 개 만들고 링크를 거는 가장 단순한 방식입니다.

<a href="login.html">로그인 페이지로</a>

방식 B 한 화면에서 숨기고 보이기 (화면 토글/SPA)

파일은 index.html 하나만 두고, 자바스크립트로 화면(섹션)을 숨겼다 보였다 하며 매끄럽게 교체하는 방식입니다. (최신 트렌드!)

document.getElementById('page1').classList.add('hidden');
document.getElementById('page2').classList.remove('hidden');

Q2. 데이터는 어떻게 저장하나요?

데이터베이스(DB)를 당장 연결하기 어렵다면, 우리 인터넷 브라우저에 내장된 개인 창고인 'LocalStorage(로컬 스토리지)'를 활용해보세요! 브라우저를 껐다 켜도 데이터가 살아있습니다.

창고에 데이터 넣기 (저장)
localStorage.setItem('username', '홍길동');
창고에서 꺼내 쓰기 (불러오기)
const name = localStorage.getItem('username');
// 화면에 '홍길동' 표시 가능!

꿀팁: MVP(최소 기능 제품) 단계에서는 복잡한 백엔드 없이 위에서 배운 방식 B(화면 토글)와 LocalStorage만으로도 훌륭한 앱 데모를 만들어낼 수 있습니다!

이번 주 팀 주간 회의 필수 미션

이번 주 팀 회의(온/오프라인)를 통해 아래 3가지를 함께 진행하고 체크해 주세요.

1. 컴포넌트 도해도 완성

  • 팀 레포지토리에 'Projects(칸반 보드)' 생성
  • 화면을 네모 박스로 쪼개고 Issue 템플릿으로 등록
  • 13주차 알파 테스트 필수 기능에 MVP 라벨 부착

2. 뼈대 세팅 및 첫 코드 PR

  • Tech Lead가 Tailwind 뼈대 파일(index.html)을 main에 업로드
  • 팀원들은 git pull 후 각자 브랜치 생성 (예: feature/header)
  • 첫 번째 MVP 컴포넌트 개발 후 GitHub에 PR 요청 올리기

3. 검토자 로컬 테스트 & 병합

  • 무작정 Merge 클릭은 절대 금지! 💣
  • 리뷰어(Tech Lead/QA)가 팀원의 브랜치를 로컬로 pull 받아 직접 화면 깨짐 확인
  • 이상 없을 시 GitHub에서 병합하고, 다 쓴 브랜치는 깔끔하게 삭제하기

Reviewer's Manual
1

로컬 컴퓨터에서 테스트

GitHub에서 눈으로만 코드를 읽고 병합하면 나중에 화면이 깨질 수 있습니다. 반드시 터미널에서 팀원의 코드를 가져와 브라우저로 띄워보세요.

# 1. 원격 저장소 최신화
$ git fetch
# 2. 팀원 브랜치로 이동 후 당겨오기
$ git checkout feature/header
$ git pull origin feature/header
브라우저에서 화면 직접 확인!
2

GitHub에서 승인 및 병합

화면에 이상이 없고 기능이 잘 작동한다면, 다시 GitHub 웹사이트로 돌아와 초록색 Merge 버튼을 눌러 원본(main)에 합쳐줍니다.

All checks have passed
1 approving review by reviewers
Merge pull request
3

다 쓴 브랜치 깔끔하게 삭제

병합이 완료된 브랜치를 방치하면 나중에 깃허브가 지저분해집니다. 웹에서 브랜치를 삭제하고 로컬 환경도 정리하세요.

Merged
# 다시 main으로 돌아와 내 로컬 정리
$ git checkout main
$ git pull origin main
$ git branch -d feature/header
