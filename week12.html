Step 2

안전한 협업을 위한 Git 브랜치 전략 🌿

여러 팀원이 동시에 코드를 작성하는 스프린트 기간에는 코드가 꼬이는 것을 방지하는 명확한 규칙이 필요합니다. GitHub Flow 기반의 가벼운 협업 방식을 소개합니다.

핵심 규칙: main 브랜치 직접 Push 지양하기

안정적인 서비스 배포를 위해 main 브랜치는 항상 동작 가능한 상태를 유지해야 합니다. 작업은 반드시 별도의 기능 브랜치(Feature Branch)에서 진행하세요.

1

이슈 기반 브랜치 생성

11주차에 생성한 이슈 번호를 활용하여 직관적인 브랜치 이름을 사용합니다.

git checkout -b feature/#이슈번호
2

Pull Request (PR) 작성

기능이 완성되면 GitHub에서 main으로 병합해 달라는 요청(PR)을 올립니다. 변경 사항을 명확히 기록하세요.

3

코드 리뷰 및 승인

담당자 외 최소 1명의 팀원이 코드를 리뷰하고 승인(Approve)한 후 안전하게 Merge 합니다.

실무에서 쓰는 브랜치 & 커밋 명명 규칙

feat/
새로운 기능 추가 시
ex) feat/#12-login
fix/
버그나 오류 수정 시
ex) fix/#15-typo
design/
CSS 등 UI/디자인 변경
ex) design/#18-button
refactor/
기능 변경 없이 코드 정리
ex) refactor/#20-cleanup

Troubleshooter
나의 사고 상황 선택

Merge Conflict (충돌) 발생

원인: 내가 작업한 파일의 라인을 다른 팀원도 수정해서 먼저 main에 올렸을 때 발생합니다.
해결책: 에러가 아닙니다! 컴퓨터가 "어느 코드가 맞는지 사람이 결정해줘"라고 묻는 것입니다. VS Code에서 조율 후 다시 커밋하세요.

bash - terminal
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
# 1. VS Code를 열어 충돌(보라색) 파일을 확인합니다.
# 2. Accept Current / Incoming 중 하나를 클릭해 코드를 정리합니다.
# 3. 정리된 파일을 다시 장바구니에 담고 커밋합니다.
$ git add .
$ git commit -m "chore: index.html 충돌 해결"
# 끝! 이제 다시 push 하시면 됩니다.
Step 3

컴포넌트 통합 및 Happy Path 검증 🧩

예쁜 화면을 넘어, 사용자가 목표를 달성하기 위해 거치는 '정상적인 사용 흐름(Happy Path)'이 막힘없이 이어지는지 확인해야 합니다.

LocalStorage로 화면 간 데이터 전달

A 화면에서 입력한 정보를 B 화면에서 보여주려면 데이터의 보관 장소가 필요합니다. 백엔드 서버 없이 MVP를 구현할 때 가장 유용한 브라우저 API인 LocalStorage를 적극 활용해 보세요.

// 1. 입력 화면 (데이터 저장)
const name = "프론트엔드";
localStorage.setItem('userName', name);
// 2. 결과 화면 (데이터 불러오기)
const savedName = localStorage.getItem('userName');
document.getElementById('user').innerText = savedName;

Happy Path 점검 예시

복잡한 예외 처리(에러 메시지 띄우기 등)는 14주차 베타 기간으로 미루어도 좋습니다. 이번 주는 정상적인 사용자가 겪는 핵심 시나리오 하나를 완벽하게 뚫어내는 것이 목표입니다.

  1. 1. 랜딩 페이지

    서비스 소개 확인 후 '시작하기' 버튼 클릭 시 라우팅 정상 작동 여부

  2. 2. 핵심 기능 입력부

    사용자 폼 입력 후 제출 시 LocalStorage에 데이터 정상 적재 확인

  3. 3. 결과 페이지 도달

    입력된 데이터를 바탕으로 추천/결과 화면이 정상 렌더링되는지 확인

Step 4

알파 테스트 대비 사전 QA (Pre-Alpha) 🐛

타 팀에게 우리 서비스를 공개하기 전, 팀원끼리 크로스 체크하여 진행을 가로막는 치명적인 버그(Blocker)를 최소화합니다.

1. 라우팅(이동) 점검

  • 누르면 아무 반응이 없는 버튼 찾기 (Dead link)
  • 이전 화면으로 돌아가는 방법(뒤로가기 등) 존재 여부

2. 콘솔(Console) 에러 확인

  • 브라우저 개발자 도구(F12) Console 탭 오픈
  • 붉은색 에러 로그가 로직 실행을 막고 있는지 파악

3. 레이아웃 붕괴 (반응형)

  • 데스크탑 브라우저 창 크기를 모바일 크기로 줄여보기
  • 콘텐츠가 화면 밖을 벗어나거나 버튼이 가려지는지 확인

발견된 주요 버그는 즉시 팀의 GitHub Issues에 Bug 라벨을 달아 기록하고 우선적으로 처리합니다.
